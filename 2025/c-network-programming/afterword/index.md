有很多人，认为 C 语言非常落后了，没有这个，没有那个。他们说，C++、Java、C#、Haskell、go、rust 以及各式各样的脚本语言，它们有这个，也有那个……其中任何一门语言都比 C 语言更现代，更值得学习。

他们的看法实际上是对的，但也是错的。在语言上，他们是对的。在编程上，他们是错的。

C 语言没有面向对象编程，面向接口编程以及函数式编程的语法，也没有异常机制、类型自动推导以及内存自动回收等特性。许多年前，我喜欢与 C++ 语言爱好者争辩，我说 C 语言也能写面向对象程序，有时会以 GObject 库作为论据，而我一直也没有真正学会这个库的用法。现在看来，争辩着实没必要。无论我有多么喜欢 C 语言，都必须坦诚地说，C 语言在设计上是很落后的。用 C 语言写程序，可视为用先秦古文的方式写文章，语言的表现力不足以编写大型软件。

在编程上，C 却比当世任何一门语言都先进。这个角度是最近这两年我才想通的。C 的先进性，通读过 CSAPP 这本书（即《深入理解计算机系统》）的人应该都能有所觉察。C 语言可嵌入汇编语言，它有指针、有线程、有进程，有进程间通信，有套接字函数，有 Win32 API，有 Unix 或 Linux 数百个系统调用，有通用计算机，也有小型嵌入式设备。

C 的先进性，一言蔽之，虚心实腹。

你可能孜孜不倦于探究面向对象领域的各种设计模式，函数式编程领域的范畴论，动态语言五彩缤纷的开发框架，在语言层面上，这些开发范式的主要工作是，提高代码的复用性。但是，软件开发的复杂度，凭借代码复用可以得到降低吗？并非如此。复杂性只是从语言层面转移到编译器或解释器的开发上了，此外语言本身变得复杂了。

复杂性，从不会消失，它只会转移。

也许你想不到的是，C 语言应对软件开发复杂度的策略是分而治之，将复杂度分割为可控的单元，然后借助线程和进程实现这些单元，再以共享内存、管道或网络套接字等通信机制将这些单元连接为系统。这种应对软件复杂度的策略从 Unix 诞生的时候便已经确立了。你不妨试着将网络编程视为一种终极的编程范式，你会发现，无论多么复杂的系统，都可以基于网络编程的方式实现，原因是，这世上还有比因特网更复杂的系统吗？

自 C++ 伊始，之后诞生的各种语言，争芳斗艳，它们努力完成的那些语法特性，对于 C 语言而言，早已尽是事实上无需多言的存在了。这些语言的爱好者，宛若孩童，跟一个饱经沧桑的老人描述他们眼中的美好世界，甚至觉得这位老人没有理由不羡慕这个世界。他不会羡慕，不过他会觉得孩童们描述的这个世界终归是可爱的，而这个世界原本就是他用了一生的时间创造出来的。
